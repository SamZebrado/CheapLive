### 数据流

```
用户打开 https://samzebrado.github.io/CheapLive/
    ↓
浏览器加载静态资源 (HTML/JS/CSS from GitHub CDN)
    ↓
从 jsdelivr 加载 MediaPipe WASM + 模型
    ↓
请求摄像头权限 → 用户授权
    ↓
MediaPipe 实时面捕 (本地推理，无服务器)
    ↓
面部参数驱动 Canvas Avatar 渲染
    ↓
(可选) 请求麦克风权限 → Web Audio 变声
    ↓
❌ 多端通信: 尝试连接 hostname:8766
    ↓
❌ 浏览器阻止 mixed-content (HTTPS→HTTP)
    ↓
❌ 无法注册设备 / 无法发现其他设备
    ↓
❌ WebRTC 无法协商
```

### 权限要求

| 权限 | 来源 | 用户交互 |
|------|------|---------|
| 摄像头 | `navigator.mediaDevices.getUserMedia()` | 需用户点击允许 |
| 麦克风 | `navigator.mediaDevices.getUserMedia()` | 需用户点击允许 |
| 通知 | `Notification.requestPermission()` | 可选 |

### 部署要求

- GitHub Pages 仓库
- 无需任何后端服务
- 无需域名
- 自动 HTTPS

### 数据流

```
设备 A (发送端)                          设备 B (接收端)
    │                                        │
    ▼                                        ▼
启动摄像头 + MediaPipe 面捕              打开接收端页面
    │                                        │
    ▼                                        ▼
向信令服务器注册 (POST /register)        订阅 SSE (GET /events)
    │                                        │
    ▼                                        ▼
获取设备列表 (GET /devices)              看到设备 A 在线
    │                                        │
    ▼                                        ▼
创建 RTCPeerConnection                    创建 RTCPeerConnection
    │                                        │
    ▼                                        ▼
生成 offer → 通过信令转发给 B            收到 offer → 生成 answer
    │                                        │
    ◄───────────────────────────────────────►
              交换 ICE candidate
    ◄───────────────────────────────────────►
              WebRTC P2P 连接建立
    │                                        │
    ▼                                        ▼
通过 DataChannel 发送面部参数            接收参数 → 驱动 Avatar
通过 AudioTrack 发送音频                 接收音频 → 播放
    │                                        │
    ▼                                        ▼
心跳保活 (5s 间隔)                       监听设备列表更新
```

### 部署要求

| 组件 | 要求 | 说明 |
|------|------|------|
| 静态页面 | 任意 HTTP 服务器 | `npx http-server . -p 8765` |
| 信令服务器 | Node.js | `node src/multi-device/signaling-server.js` |
| 网络 | 同一 LAN | 设备需在同一个 WiFi |
| HTTPS | 可选但推荐 | 如果不使用摄像头/麦克风，HTTP 即可 |

### 数据流

```
用户打开 Android App
    ↓
启动前台服务 (显示通知: "CheapLive 正在运行")
    ↓
请求悬浮窗权限 → 用户手动授权
    ↓
创建悬浮窗 (透明背景 + WebView)
    ↓
WebView 加载 CheapLive 页面
    ↓
请求摄像头权限 → 用户授权
    ↓
MediaPipe 面捕运行
    ↓
Avatar 渲染在悬浮窗中
    ↓
用户可切换到其他 App
    ↓
悬浮窗保持显示 (始终置顶)
    ↓
(限制) 锁屏后摄像头停止，悬浮窗可能被系统关闭
```

### 原生层要求

| 功能 | 原生代码要求 | 难度 |
|------|-------------|------|
| 前台服务 | Android Service + Notification | 中 |
| 悬浮窗 | WindowManager + SYSTEM_ALERT_WINDOW | 中 |
| 触摸穿透 | FLAG_NOT_TOUCHABLE | 低 |
| 始终置顶 | TYPE_APPLICATION_OVERLAY | 低 |
| 后台摄像头 | 前台服务 + CameraX | 高 |
| WebRTC | WebView 或 Chrome Custom Tabs | 中 |

## 关键结论

1. **GitHub Pages 单端模式**是最可靠的 Demo 方案，但无法展示多端通信
2. **LAN 本地多端模式**需要用户同时运行 Node 信令服务器，增加了演示门槛
3. **Android WebView 悬浮模式**是最终产品形态，但开发工作量大，不适合比赛 Demo
4. **Live2D 模式**技术上可行，但当前代码差距大，且受许可证约束
