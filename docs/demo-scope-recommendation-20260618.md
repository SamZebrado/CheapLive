**生成日期**: 2026-06-18

## 1. 核心结论

### Live2D

| 问题 | 答案 |
|------|------|
| 当前网页能否加载真正 Live2D? | **不能**。代码仅为骨架，无渲染循环，无模型资源 |
| 普通网页能否加载 Live2D? | **能**。技术上可行，已有开源项目证明 |
| GitHub Pages 能否加载 Live2D? | **能**。静态 HTTPS 满足要求，但模型文件需用户自有 |
| Demo 是否建议暂缓 Live2D? | **是**。当前代码差距大，许可证有约束，程序化 Avatar 已够用 |

### 纯 Web 最大能力边界

- ✅ 实时面捕 + 程序化 Avatar 渲染
- ✅ 本地音频处理 (变声)
- ✅ WebRTC P2P (同一 LAN)
- ❌ 无后端 = 无设备注册/发现
- ❌ 锁屏后摄像头停止
- ❌ 系统悬浮窗
- ❌ 后台持续运行

### GitHub Pages 最大能力边界

- ✅ 单端面捕 + Avatar
- ❌ 多端通信 (mixed-content 阻止 HTTP 信令)
- ❌ 本地 Node 服务
- ❌ UDP/LAN 扫描

### 必须依赖本地服务的功能

- 多端设备注册与发现 (HTTP/SSE 信令服务器)
- WebRTC 初始协商 (offer/answer 交换)
- 任何需要服务器状态的功能

### 必须依赖 Android 原生层的功能

- 系统透明悬浮窗
- 始终置顶
- 触摸穿透
- 前台服务 (防杀)
- 后台摄像头
- 跨应用叠加

## 3. Demo 功能范围建议

### Demo 必须保留 (核心展示)

这些功能在当前代码中已实现，且在 GitHub Pages 上可直接运行，是比赛 Demo 的核心卖点:

1. **实时面部捕捉**
   - MediaPipe Face Landmarker
   - 52 Blendshape + head pose
   - 前后摄像头切换
   - 纯浏览器端推理，无后端

2. **程序化 Avatar 渲染**
   - 球形头像 (带五官)
   - 纺锤鲸鱼 (带尾巴摆动)
   - 眨眼、张嘴、微笑、眉毛驱动
   - 头部转向 (yaw/pitch/roll)
   - 自动取景

3. **交互控制**
   - 镜像模式
   - 隐私模式 (停止摄像头)
   - 性能模式 (流畅/省电/极简)
   - 灵敏度调节

4. **本地配置**
   - localStorage 持久化
   - Avatar 切换
   - 参数记忆

### Demo 可选 (增强体验)

这些功能可以增加 Demo 的丰富度，但不是核心:

1. **屏幕常亮** (Wake Lock API)
   - 代码简单，Baseline 2025
   - 适合长时间演示

2. **全屏模式**
   - Fullscreen API 成熟
   - 提升沉浸感

3. **PWA 安装提示**
   - 添加 manifest.json 和 Service Worker
   - 让用户可以"安装"到桌面

4. **通知**
   - 连接状态通知
   - 低优先级

### Demo 降级 (降低期望)

这些功能代码存在，但存在已知限制，应在 Demo 中明确标注:

1. **多端通信**
   - **GitHub Pages 上不可用** (mixed-content)
   - 仅在本地 LAN 演示时可用
   - 建议改为"手动输入对方 ID"而非自动发现
   - Demo 话术: "同一 WiFi 下可 P2P 连接，需本地启动信令服务"

2. **变声器**
   - 代码存在但生产集成未验证
   - ScriptProcessorNode 已弃用
   - Demo 话术: "实验性功能，音质有限"

3. **音频推流**
   - 真实跨设备未验证
   - Demo 话术: "技术验证阶段"

### Demo 暂时隐藏

这些功能在 UI 中存在但无法实际工作，会造成用户困惑:

1. **Live2D 自定义模型**
   - 当前代码无法渲染
   - 选择后会报错
   - **建议**: 从下拉菜单中移除，或改为灰色不可选并标注"开发中"

2. **Live2D 上传入口**
   - 有 UI 但无后端渲染
   - **建议**: 隐藏或标注"即将支持"

3. **字幕功能**
   - 如果 UI 中有开关但无实现
   - **建议**: 隐藏

### 后续路线图 (未来开发)

这些功能不应出现在 Demo 中，但可作为未来规划:

1. **Android 原生 App**
   - 悬浮窗
   - 前台服务
   - 后台摄像头
   - 这是产品的最终形态

2. **Live2D 完整支持**
   - 完成 Cubism 渲染循环
   - 解决 SDK 许可证
   - 提供默认模型

3. **高质量变声**
   - 迁移到 AudioWorklet
   - 更好的算法

4. **云端信令**
   - 部署到免费云服务器 (如 Render, Railway)
   - 解决 GitHub Pages 的 mixed-content 问题
   - 实现真正的"开箱即用"多端通信

## 4. 三种独立演示模式建议

### 模式一: GitHub Pages 单端 Demo (主推)

**用途**: 比赛线上评审、用户快速体验
**链接**: `https://samzebrado.github.io/CheapLive/`
**展示内容**:
- 打开页面 → 允许摄像头 → 立即看到 Avatar 跟随面部
- 测试眨眼/微笑/张嘴按钮
- 切换球形/鲸鱼 Avatar
- 调节灵敏度

**优势**:
- 零安装
- 零配置
- 任何设备打开即用
- 最稳定的展示路径

### 模式二: 本地 LAN 多端 Demo (现场展示)

**用途**: 比赛现场答辩、线下演示
**准备**:
- 一台电脑运行 `node src/multi-device/signaling-server.js`
- 所有设备连接同一 WiFi
- 访问电脑的 LAN IP

**展示内容**:
- 设备 A 作为发送端 (摄像头 + 面捕)
- 设备 B 作为接收端 (显示 Avatar)
- 展示 P2P 参数同步

**注意**:
- 需提前测试网络环境
- 准备备用方案 (如果 WiFi 不稳定)
- 明确告知评委"需要本地服务"

### 模式三: Android 悬浮窗 Demo (概念展示)

**用途**: 展示产品最终愿景
**形式**:
- 视频录制或截图
- 展示"悬浮在屏幕上的虚拟形象"
- 说明"需要原生 Android 开发"

**注意**:
- 当前无实际代码
- 仅用来说明未来方向
- 不要声称已实现

## 5. 对外介绍建议口径

### 必须说的 (核心卖点)

> "CheapLive 是一个纯浏览器端的低成本虚拟形象面部捕捉工具。用户无需安装任何 App，打开网页即可使用摄像头驱动 3D 虚拟形象，支持眨眼、张嘴、微笑等表情，以及头部转向。"

### 可以说的 (扩展能力)

> "在同一 WiFi 环境下，可以通过本地服务实现多端 P2P 连接，将面捕数据实时同步到另一台设备。"

### 必须避免说满的 (当前限制)

| 避免说 | 改为 |
|--------|------|
| "支持 Live2D" | "计划支持 Live2D，当前使用程序化 Avatar" |
| "支持多端自动发现" | "同一 WiFi 下可手动连接，自动发现需本地服务" |
| "支持系统悬浮窗" | "未来计划开发 Android 原生版实现悬浮窗" |
| "支持后台运行" | "浏览器安全限制，锁屏后摄像头停止" |
| "支持变声推流" | "变声功能实验性，真实推流待验证" |
| "零后端" | "单端模式零后端，多端模式需本地信令服务" |

### 建议的 Demo 话术

**开场**:
> "这是一个纯网页的虚拟形象面捕工具。我现在打开这个链接，你们可以看到——不需要安装任何东西，直接就能用。"

**面捕展示**:
> "我允许摄像头权限，现在我的虚拟形象会实时跟随我的表情。我眨眼，它眨眼；我张嘴，它张嘴；我转头，它也会转头。"

**多端展示 (如果现场有 LAN)**:
> "如果在同一 WiFi 下，我还可以启动一个本地服务，让另一台设备同步显示我的 Avatar。这是纯 P2P 的，数据不经过任何服务器。"

**收尾**:
> "当前是网页版，未来我们计划开发 Android 原生版，实现系统悬浮窗，让用户可以在使用其他 App 时，虚拟形象始终显示在屏幕上。"

## 6. 红米 K50 / QQ 浏览器 风险提醒

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| QQ 浏览器限制 WebRTC | 多端通信不可用 | 使用 Chrome 演示 |
| 微信内置浏览器限制文件上传 | Live2D 上传不可用 | 不使用微信浏览器 |
| MIUI 后台限制 | 切后台后面捕停止 | 保持页面在前台 |
| WebGL 性能 | 复杂 Avatar 可能卡顿 | 使用性能模式 |
| 摄像头权限弹窗 | 首次使用需授权 | 提前告知评委 |

**建议**: 比赛 Demo 使用 Chrome Android 或 Safari iOS，避免 QQ 浏览器和微信内置浏览器。

## 8. 总结

| 类别 | 数量 | 功能 |
|------|------|------|
| Demo 必须保留 | 4 类 | 面捕、Avatar、交互控制、本地配置 |
| Demo 可选 | 4 项 | Wake Lock、全屏、PWA、通知 |
| Demo 降级 | 3 类 | 多端通信、变声、音频推流 |
| Demo 隐藏 | 3 项 | Live2D 选项、上传入口、字幕 |
| 后续开发 | 7+ 项 | Android 原生、Live2D、云端信令等 |

**核心建议**:
1. **GitHub Pages 单端 Demo 是比赛的主展示路径**
2. **多端通信作为现场加分项，但需提前准备环境**
3. **Live2D 从 UI 中隐藏，避免用户困惑**
4. **所有"未来功能"用"计划"而非"支持"来描述**
