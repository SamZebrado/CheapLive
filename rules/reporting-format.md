# 汇报格式与禁止性表述

本文件定义所有 Agent 的汇报格式、禁止性表述和未验证项保留要求。

## 1. 核心原则

- 证据优先，结论从严
- 区分不同层级的完成度（想法成立 / 代码已写 / 静态检查通过 / 本地命令跑通 / 端到端验证通过 / 项目可交付）
- 宁可少说，不可说满
- 任何正面结论都必须附带边界

## 2. 推荐汇报格式

以后状态汇报默认按这个结构写：

### 2.1 本轮实际完成

```text
- [已修改] 调整了 A 模块中的输入校验逻辑。
- [已修改] 统一了 B 和 C 两个入口的参数处理。
- [已修改] 收紧了 D 场景下的错误处理。
```

### 2.2 证据

```text
- [已阅读] 已查看相关代码与配置文件。
- [已核对] 当前目标版本中可看到上述修改。
- [已运行静态检查] 已运行语法检查与构建前检查，结果通过。
- [已运行本地测试] 已运行最小功能测试，主链路通过。
```

### 2.3 当前不能下结论的部分

```text
- [未验证] 真实用户环境中的端到端表现。
- [未验证] 高负载或异常输入下的稳定性。
- [未验证] 打包产物是否已被第三方独立复核。
```

### 2.4 当前最准确状态

```text
当前版本已完成这轮代码修改，并通过了局部检查；但尚不能下结论说项目已整体可用或可直接交付。
```

### 2.5 下一步建议

```text
先完成指定环境下的端到端验证，再决定是否宣布可用或交付。
```

## 3. 禁止绝对化表述

以下词语默认禁止使用，除非同时给出严格证据和明确验收依据：

- 完全
- 全部
- 所有
- 一定
- 肯定
- 已确保
- 已保证
- 没有问题
- 已修复
- 已解决
- 已完成
- 已准备好
- 可以直接交付
- 功能正常
- 通过测试
- 稳定可用
- 最终版
- 已彻底解决
- 没有风险

如果证据不够，改成：

- 已实现
- 已添加
- 已调整
- 已收紧
- 已在代码中看到
- 已在本地修改
- 已通过静态检查
- 已通过部分测试
- 当前版本中已完成
- 降低了……风险
- 覆盖了当前已知场景
- 尚待进一步验证
- 目前可以确认的是……
- 目前不能确认的是……

## 4. 测试表述规则

### 4.1 禁止只写"通过测试"

必须写清楚：

- 测了什么
- 用什么方式测的
- 测试范围是什么
- 没覆盖什么

### 4.2 静态检查不能包装成真实可用

以下结果都不能直接推出"功能正常"或"项目可交付"：

- 语法通过
- lint 通过
- 类型检查通过
- 构建通过
- smoke test 通过
- 单元测试通过

### 4.3 本地跑通不能自动等于用户环境可用

必须明确区分：

- 本地环境跑通
- 指定环境跑通
- 用户真实环境跑通

### 4.4 没做端到端验证就不能写

- 功能正常工作
- 可以直接使用
- 已准备好交付
- 生产可用
- 没问题了

测试汇报必须写成：

```text
- [已运行静态检查] 已运行语法检查，结果通过。
- [已运行本地测试] 已运行最小功能脚本，结果通过；仅覆盖单一输入场景。
- [未验证] 尚未完成真实环境端到端验证。
```

## 5. 安全、性能、兼容性表述规则

### 5.1 安全类表述必须限定范围

不要写："安全问题已解决。"

要写："已对 X 环节增加 Y 校验，从而降低 Z 风险。"

### 5.2 性能类表述必须有依据

不要写："性能更好了。"

要写："在当前测试环境下，A 操作耗时从 X 降到 Y；尚未评估其他场景。"

### 5.3 兼容性类表述必须限定平台和范围

不要写："已兼容各种环境。"

要写："已在 A 环境和 B 环境下验证；其他环境未验证。"

### 5.4 只做了局部优化禁止外推成整体结论

- 某个路径更安全，不等于整体安全通过
- 某个页面可用，不等于整个产品稳定
- 某个脚本通过，不等于整套流程无问题

## 6. 产物与工作区一致性规则

必须区分以下对象：

- 本地工作区
- 当前分支
- 当前提交
- 构建产物
- 打包文件
- 用户实际拿到的附件
- 审核者实际看到的文件

如果没有亲自检查产物，就不能写：

- 当前附件已经包含这些修改
- 当前包就是最新版本
- 用户拿到的就是我刚改好的版本

如果修改只存在于工作区，还没有重新打包或重新构建，就必须明确写：

- 修改已在本地完成，但产物尚未重新生成或复核

## 7. 可用、ready、可交付的使用门槛

只有当以下条件中与项目相关的部分都满足时，才能使用"可用 / ready / 可交付 / 可以开始使用 / 可以上线"这类说法：

1. [已核对] 相关修改已存在于当前目标版本中
2. [已运行静态检查] 基础检查通过
3. [已运行本地测试] 关键主链路至少跑通一次
4. [已运行集成测试] 相关模块联动已验证（若项目需要）
5. [已运行端到端测试] 真实或近真实流程已验证（若项目需要）
6. 已知失败项、限制项、未验证项已明确列出
7. 当前结论的适用环境已说明清楚

只要缺关键一项，就不能写：

- 已准备好使用
- 可以直接交付
- 功能已经正常工作
- 项目已经完成
- 可以放心上线

更准确的替代说法：

- 当前代码修改已完成，仍待验证
- 当前版本通过了局部检查，尚不能下结论说整体可用
- 当前产物已具备继续验收的条件

## 8. 未验证项必须保留

每次总结末尾必须固定增加一节：

```text
未验证项：
- …
- …
- …
```

哪怕本轮已经做了很多工作，也必须保留这一节。不要为了让总结更好看而删掉未验证项。

常见未验证项包括：

- 真实环境验证
- 边界条件
- 异常路径
- 大数据量/高并发场景
- 跨平台兼容性
- 外部依赖变更后的表现
- 用户实际操作路径
- 打包产物与源码一致性

## 9. 禁止性句式

以下句式默认禁止：

- "不过没关系，我们已经全部搞定了"
- "现在没有问题了"
- "已经可以直接用了"
- "项目已经准备好了"
- "质量已经保证"
- "功能正常工作"
- "通过测试，基本没问题"
- "所有必要修复都已完成"
- "已经彻底解决"
- "可以放心交付"
- "应该不会有问题"
- "我确认没问题"

请改成更实际的句式：

- "我已完成以下修改，但仍有未验证项。"
- "当前可以确认的是……，尚不能确认的是……"
- "已通过局部检查，不等于真实环境已验证通过。"
- "已降低相关风险，但不等于整体问题已彻底解决。"
- "当前版本已具备继续验收的条件，是否可交付仍待验证。"

## 10. 面对不确定性时的正确做法

如果遇到以下情况，必须主动降调结论：

- 没有运行测试
- 测试只覆盖局部
- 没看最新产物
- 没看用户实际附件
- 没有真实环境
- 没有权限验证外部系统
- 依赖环境和用户环境不同
- 有关键路径仍靠推测

此时正确说法只能是：

- 不确定
- 未验证
- 尚待确认
- 只能确认局部
- 当前无法判断整体结论

## 11. 测试矩阵（最终报告必须包含）

每次开发完成后的最终报告必须包含以下测试矩阵，未执行项不能省略：

| 层级 | 对象 | 是否生产代码 | 结果 | 未执行原因 |
|------|------|-------------|------|-----------|
| 静态检查 | import/字段/路径 | 是 | | |
| 单元测试 | 辅助函数 | 是 | | |
| 实例化 | 生产 class | 是 | | |
| 集成 | 跨模块调用 | 是 | | |
| 页面 | 真实 HTML | 是 | | |
| Android | build/Activity | 是 | | |
| 真机 | 实体设备 | 是/否 | | |
| 视觉 | 用户验收 | 否 | | |

填写规则：

- "结果"列填写：通过 / 失败 / 跳过 / 未执行
- "未执行原因"列：未执行的行必须填写原因（如"无真机在线"、"本轮不涉及 Android"）
- 禁止留空未执行行而不写原因
- 禁止用高等级结果替代低等级（如"实例化通过"不能替代"单元测试通过"）
