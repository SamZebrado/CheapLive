# Runtime Validation and Completion Gate

本文件定义开发Code在提交最终完成报告前必须通过的强制验证层级。这些规则适用于以后所有 CheapLive 开发任务，不限于当前修复。

## 1. 测试由开发Code主动设计

开发Code不能只运行用户明确点名的测试或 Prompt 中列出的测试。

每次修改生产代码后，必须根据真实 diff 主动识别以下风险并补充测试：

- 生产入口（HTML 页面、Activity、main 函数）
- 真实实例化对象（生产 class、service、renderer、bridge、controller）
- 构造顺序（父子类初始化、依赖注入时序）
- 重要状态分支（mirror / app mode、连接状态、权限状态）
- 跨文件契约（字段名、类型、方向、单位）
- 坐标和单位（X/Y/Z 方向、角度零点、正方向）
- 异步时序（Promise 链、事件顺序、生命周期回调）
- 页面或 Activity 入口（真实加载、真实启动）
- 最容易失败的场景（边界值、空输入、快速连续操作）

## 2. 三层最低 Gate

每次开发至少经过以下三层，缺一不可判定完成：

```
静态与结构检查
真实生产对象实例化
真实用户入口 smoke test
```

### 2.1 静态与结构检查

至少检查：

- import/export 路径和存在性
- 动态 import 路径可解析性
- 字段和方法存在性（拼写一致）
- DOM id 与代码引用一致
- 类构造参数与调用处匹配
- 父子类初始化顺序正确
- 事件名与监听处匹配
- 参数名在定义和调用间一致
- 生成文件有明确来源和生成命令
- 残留旧名称已清理
- Git diff 异常规模（小任务不应产生数百行改动）

### 2.2 真实对象实例化

本轮修改的生产 class、service、renderer、bridge 或 controller 必须真实实例化，并执行关键方法。

禁止以下替代方式判定通过：

- 只测试辅助函数
- 只检查字符串是否存在
- 在测试中复制生产类并测试副本
- 只返回固定值的 mock

Mock 只允许用于以下边界，不得替代被测生产对象：

- 浏览器 API（camera、microphone、DOM）
- 音频 API（AudioContext、MediaStream）
- 网络请求（WebSocket、fetch）
- 系统边界（文件系统、设备传感器）

### 2.3 真实入口 smoke test

网页功能必须打开真实 HTML 并加载真实生产脚本，确认页面无报错。

Android 功能必须至少完成：

```
testDebugUnitTest
assembleDebug
Activity launch smoke（adb 启动并检查 logcat 无崩溃）
```

设备在线时，合理情况下继续安装、启动和检查 logcat。

## 3. Bug 回归测试要求

每个 bug 修复至少有一个回归测试，同时满足：

- 旧实现会失败
- 新实现会通过

具体场景的最低要求：

- 初始化顺序问题：必须真实构造子类并验证字段值
- 属性拼写错误：必须进入对应分支并断言正确值
- 坐标系错误：必须断言具体方向和数值（如中性姿态法线）
- 尺寸闪烁：必须连续采样多帧并断言稳定性
- 生命周期问题：必须执行暂停和恢复并验证状态
- 连接问题：必须检查真实连接状态，而非仅检查元素存在

## 4. 跨文件契约检查

凡数据跨越两个以上文件或运行环境，开发Code必须建立并核对契约：

```text
字段名
来源
接收方
类型
范围
默认值
中性值
单位
方向
错误处理
```

需要覆盖可能相关的环境：

```text
Web UI（HTML/JS）
JavaScript 生产逻辑
Android/Kotlin
JavaScript Bridge
网络协议（WebSocket 消息）
派生脚本（自动生成文件）
自动测试
README / 文档
```

字段名相同但语义不同，视为契约失败，必须重命名或明确区分。

## 5. 几何和坐标系 Gate

涉及 Canvas、2D/3D、网格、姿态、投影和旋转时，代码和测试必须明确：

```text
X 轴方向
Y 轴方向
Z 轴方向
摄像机方向
正面方向
角度零点
角度正方向
旋转顺序
参数范围
中性值
```

测试必须使用实际数值断言，不得只检查"发生变化"。

最低断言要求：

- 中性姿态下主要五官可见
- 正面锚点朝向摄像机
- 左右五官位于同一近侧面
- 嘴巴在眼睛下方
- 远侧五官随转向压缩或隐藏
- 锚点不超出身体边界
- 投影尺寸在合理范围内保持稳定

## 6. 生成文件 Gate

所有生成或同步文件必须：

- 有单一源文件
- 有明确生成命令
- 可以重复生成
- 重复运行后 `git diff` 为空
- 禁止手工维护两个派生副本
- 对相同输出做 hash 或内容一致性检查

## 7. 提交前 diff review

commit 前必须执行并阅读：

```bash
git status --short
git diff --check
git diff --stat
git diff
```

主动检查：

- 小任务是否意外重写大量代码
- 是否删除既有功能
- 是否引入重复实现
- 是否提交缓存、build 或机器生成文件
- 是否有未跟踪的重要文件
- 最终总结是否符合 diff 实际内容

小修出现数百行以上改动时，开发Code必须解释原因并增加回归覆盖。

## 8. 反驳自己检查

最终汇报前，开发Code必须主动尝试推翻自己的"完成"结论，逐项检查：

- 是否只测了辅助函数？
- 是否未真实实例化生产对象？
- 是否有分支从未执行？
- 是否只在 mock 中通过？
- 是否没有打开真实页面？
- 是否把 build 成功当作运行成功？
- 是否把截图生成当作视觉通过？
- 是否把 push 成功当作 Pages 部署成功？
- 是否把真机待验证问题藏在"完成"之后？

发现基础集成问题时，应先修复，不得只放入未验证清单。

## 9. 灵活性边界

本规则约束验证质量，不硬编码以下内容：

- 固定测试框架
- 固定函数名
- 固定文件布局
- 每次都必须使用 Playwright
- 每次都必须连接真机
- 每个小改动都跑全部耗时测试

开发Code可根据修改范围选择工具，但必须覆盖：

```text
真实生产对象
真实调用链
真实入口
关键失败模式
```

纯文档修改可免除运行时 Gate，但仍需做链接、状态一致性和 diff 检查。

## 10. 开发Work 权限例外

本文件及 AGENTS.md、rules/ 下的规则文件由开发Work维护，属于规则文档维护例外：

- 开发Work可编辑 AGENTS.md 和 rules/
- 开发Work不得修改 src/、tests/、android-capture/、构建配置
- 开发Work不得 commit 或 push
- 开发Code负责审阅规则变更并提交
- 开发Code不得在规则审阅时顺便修改产品功能
- 同一时间不得由两个 Agent 修改同一个规则文件
