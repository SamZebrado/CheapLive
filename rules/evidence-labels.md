# 证据标签与成果防夸大规则

本文件定义 CheapLive 项目中所有 Agent 必须使用的证据标签，以及防止成果夸大的具体规则。

## 1. 证据标签（必须使用）

以后所有总结和状态汇报，都必须给关键结论加证据标签。只允许使用以下标签：

| 标签 | 含义 |
|------|------|
| `[已阅读]` | 实际读过相关代码、文档、配置、日志、数据、文件 |
| `[已修改]` | 亲自完成了修改，但还没有被独立复核 |
| `[已核对]` | 在当前版本、当前文件、当前产物中重新确认过相关内容存在 |
| `[已运行静态检查]` | 实际运行了语法检查、lint、类型检查、构建前检查、格式检查、静态分析等 |
| `[已运行本地测试]` | 实际运行了本地命令、脚本、单元测试、最小功能测试，并看到了输出结果 |
| `[已运行集成测试]` | 实际完成了多模块联动验证，但不一定是真实用户环境 |
| `[已运行端到端测试]` | 在真实或近真实环境中完成了完整流程验证 |
| `[未验证]` | 没有足够证据，或尚未进行验证 |

禁止自造模糊标签，如 `[已确认]`、`[已保证]`、`[已通过]`、`[已就绪]`、`[应该没问题]`。除非后面紧跟明确证据，否则一律不用。

## 1.1 证据等级（不可互相替代）

以下等级存在严格递进关系，低等级不得替代高等级：

```text
语法通过
辅助函数测试通过
生产对象实例化通过
真实入口自动测试通过
真实设备验证通过
用户视觉验收通过
```

- "语法通过"不等于"功能可用"
- "辅助函数测试通过"不等于"生产对象实例化通过"
- "生产对象实例化通过"不等于"真实入口自动测试通过"
- "真实入口自动测试通过"不等于"真实设备验证通过"
- "真实设备验证通过"不等于"用户视觉验收通过"

在汇报中必须明确标注当前达到的最高等级，不得跳级表述。

## 2. 成果防夸大规则

### 2.1 代码存在不等于功能接入

出现以下情况时只能写"代码或入口已存在"，不能写"功能已接入"：

- 创建了 class
- 添加了 loader
- 增加了接口
- 写了骨架
- 添加了 UI
- 添加了配置项

除非生产调用链实际运行，才能写"功能已接入"。

### 2.2 测试存在不等于功能通过

必须检查测试是否：

- 测试生产代码（而非复制生产类到测试中）
- 通过真实 UI
- 使用真实状态
- 具有有效断言（没有固定 `true`）
- 没有只检查 DOM、字符串或元素存在
- 没有用宽松条件放过错误
- 没有把 flaky 算作稳定通过
- 没有通过删除困难测试换取绿色结果

### 2.3 截图生成不等于视觉通过

只能写：

```
视觉材料已生成，等待用户验收
```

### 2.4 浏览器自动化不等于真机通过

必须明确区分：

- 无头 Chromium
- 桌面浏览器
- 手机浏览器
- 红米 K50
- QQ 浏览器
- Android WebView
- 真实跨设备局域网

### 2.5 信令通过不等于 WebRTC connected

必须检查：

```
connectionState
iceConnectionState
ontrack
DataChannel open
```

### 2.6 音频 track 存在不等于变声推流完成

必须确认：

```
麦克风输入
→ VoiceChanger 生产实现
→ processed stream
→ RTCRtpSender
→ receiver ontrack
```

## 3. 固定状态词汇

统一使用：

```
已实现并稳定自动测试
已实现但验证不完整
部分实现
仅骨架/占位
未实现
真实设备待验证
用户视觉验收待确认
受资产或许可证阻塞
```

避免：

```
基本完成
全面支持
生产级
完美运行
完整闭环
```

除非有充分证据。

## 4. 事实、推断与决策的区分

### 4.1 已确认事实

来自：

- 当前 GitHub 代码
- 实际命令输出
- 测试 reporter
- 官方文档
- 用户明确决定

### 4.2 合理推断

必须写明：

```
推测 / 可能 / 需要验证
```

### 4.3 产品决策

只有用户明确决定后才能写为既定路线。不得把"我建议"偷偷改写为"项目已经决定"。
