# CheapLive Agent Rules

本文件是 CheapLive 项目的最高优先级开发规则。

任何 TRAE Work、TRAE IDE 或其他获准参与项目的 Agent，在开始任务前必须阅读本文件。

## 1. 项目核心目标

CheapLive 的核心目标是：

> 为没有电脑或设备性能有限的个人主播，提供支持单端和多端组合的移动 Live2D 面捕与直播辅助工具。

核心价值是降低设备、软件、性能和学习门槛。

## 2. 当前产品主线

开发优先级严格按照以下顺序：

1. Live2D 模型基础显示；
2. 透明悬浮显示；
3. 手动表情与动作控制；
4. 单端基础面捕；
5. 单端性能控制；
6. 双端局域网面捕；
7. 双端直播制作；
8. 可选实时变声。

不得为了后续功能，阻塞或破坏前面的核心闭环。

## 3. 变声器不是当前 MVP

实时变声属于备选功能。

在以下功能稳定前，不得把主要开发时间投入变声器：

* Live2D 模型可以稳定加载；
* 模型可以透明悬浮；
* 基础表情和动作可以触发；
* 单端面捕可以运行；
* 单端模式具备基础性能设置。

## 4. 不允许擅自改变产品方向

Agent 不负责决定是否：

* 改变赛道；
* 改变目标用户；
* 更换核心产品定位；
* 增加云端账户；
* 增加模型商店；
* 开发 iOS 版本；
* 建设完整直播平台；
* 重写整个现有悬浮浏览器。

这些决策必须由项目负责人明确授权。

## 5. 优先复用现有悬浮浏览器

现有悬浮浏览器是 CheapLive 的基础能力。

开发新功能前必须确认：

* 当前悬浮窗口功能是否正常；
* 多窗口功能是否正常；
* 触摸穿透和窗口锁定是否正常；
* Foreground Service 是否正常；
* 现有配置和数据是否能够保留。

不得在没有迁移方案和备份的情况下删除现有实现。

## 6. 单端模式必须始终可用

即使双端功能尚未完成，CheapLive 也必须保留一台手机即可运行的模式。

双端模式用于降低性能压力，不能成为使用产品的强制条件。

## 7. 双端模式优先传输参数

双端轻量面捕模式优先传输：

* 眼睛开合参数；
* 嘴部开合参数；
* 头部姿态；
* 表情编号；
* 动作指令；
* 必要的连接状态。

第一阶段不要传输完整透明视频流。

局域网通信必须具备：

* 明确的连接状态；
* 配对机制；
* 断线提示；
* 重连机制；
* 用户主动关闭入口。

## 8. 摄像头数据默认本地处理

禁止默认：

* 上传摄像头画面；
* 保存摄像头视频；
* 上传面部特征；
* 在用户不知情时后台采集。

多端模式只在用户自己的局域网内传输必要参数。

任何需要改变此原则的功能都必须获得明确授权。

## 9. 开源与模型授权必须可追溯

引入第三方依赖前必须记录：

* 项目名称；
* 官方地址；
* 当前版本；
* 许可证；
* 使用目的；
* 是否允许商业使用；
* 是否要求保留声明；
* 是否存在模型格式或 SDK 的额外限制。

不得仅因为代码可以下载，就默认可以自由用于发布版本。

用户导入的模型必须提示其自行确认直播、展示和商业使用授权。

## 10. 每个任务必须是原子任务

任务应包含：

* 明确目标；
* 修改范围；
* 禁止修改范围；
* 验收标准；
* 验证命令；
* 交付格式。

不得把架构重构、新功能、UI 改版和发布准备混在一个任务中。

## 11. 不得虚构完成状态

只有在存在对应证据时，才能使用以下表述：

* 已构建成功；
* 已安装成功；
* 已在设备运行；
* 已通过测试；
* 已验证功能。

如果只阅读了代码，应写：

* 静态检查未发现问题；
* 根据代码逻辑推测；
* 尚未进行运行验证。

## 12. 每次完成报告必须包含

1. 修改文件；
2. 修改内容；
3. 执行命令；
4. 测试结果；
5. 未测试内容；
6. 已知风险；
7. Git 状态；
8. 下一项原子任务。

## 13. 不要做无动作等待

任务完成后直接报告。

任务阻塞时报告：

* 阻塞原因；
* 错误原文；
* 已尝试方案；
* 当前最小下一步。

不要反复输出没有新内容的总结。

## 14. 不允许无关修改

未经授权不得：

* 全仓库格式化；
* 批量改名；
* 升级无关依赖；
* 修改包名；
* 修改应用名称；
* 删除历史兼容代码；
* 改写与当前任务无关的文档。

## 15. Git 操作规则

* 默认不自动 Push；
* 默认不强制 Push；
* Commit 前检查 diff；
* 一个 Commit 对应一个原子目标；
* 大规模修改前创建备份分支；
* 不覆盖仍可运行的稳定版本；
* 未经授权不修改历史 Commit。

## 16. 性能结论必须有依据

性能报告至少写明：

* 设备；
* Android 版本；
* 运行模式；
* 测试时间；
* Live2D 帧率；
* 面捕频率；
* 是否同时运行游戏和直播；
* 可观察到的问题。

没有测量时，不得声称已经显著优化性能。

## 17. TRAE Work 与 TRAE IDE 分工

TRAE Work 优先负责：

* 产品需求；
* 用户流程；
* 页面方案；
* 任务拆解；
* 文档；
* 阶段验收。

TRAE IDE 优先负责：

* 仓库检查；
* 代码修改；
* 构建；
* 安装；
* 设备测试；
* 错误修复；
* Git 状态报告。

不同 Session 之间必须通过项目文件和结构化交接继续工作，不依赖未保存的聊天记忆。

## 18. 参赛证据保留

开发过程中必须保留：

* 关键 Session ID；
* 关键任务截图；
* 阶段结果截图；
* 构建记录；
* 测试记录；
* Git Commit；
* 第三方依赖记录。

不得在项目完成后临时伪造或补写开发过程。

## 19. 冲突处理

如果当前任务与本文件冲突：

1. 暂停冲突部分；
2. 明确指出冲突条款；
3. 完成不冲突的部分；
4. 等待项目负责人给出新的明确指令。

## 20. 最终原则

优先保证：

1. 可运行；
2. 可验证；
3. 不破坏现有功能；
4. 隐私安全；
5. 授权清晰；
6. 范围可控；
7. 演示完整。

不要为了功能数量牺牲核心闭环的稳定性。
